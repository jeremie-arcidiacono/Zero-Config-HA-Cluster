### Investigation k3s server/agent

#### Nombre pair de servers est un vrai problème ?

Oui.

Un nombre pair augmente le nombre de node de 1 (donc une nouvelle node susceptible de tomber), sans pour autant
augmenter la tolérance aux pannes.  
Donc le cluster devient plus fragile.

Lorsque le quorum n'est pas atteint, etcd passe en mode lecture seule, ce qui rend le cluster inutilisable.
Dans le cas d'un split brain avec 2 groupes équivalents, le cluster devient inutilisable.

#### Transition d'un server en agent et inversement

Pas de manière officielle et "simple" de le faire.  
Les rôles dans Kubernetes ne sont en réalité que des labels et des configurations appliquées aux binaires exécutés.

Scale up : convertir un agent en server (plus simple en théorie)
Scale down : convertir un server en agent

Process :

1. arrêt propre du k3s en cours (avec un drain)
2. supprimer le node dans le cluster `kubectl delete node <node-name>`
3. démarrer k3s avec la nouvelle commande (`k3s agent` ou `k3s server`)

Durant scale down, il faut vraiment s'assurer que le node est supprimé avant de le convertir.  
Cela demande peut-être une étape supplémentaire dans etcd : apparemment, `kubectl delete` doit être accompagné d'un
`etcdctl member remove`, sinon etcd garde le node.

2 exemples :

##### Situation 1 : 5 machines (5 servers), puis 1 tombe en panne

Si la panne dure, on doit scale down, afin de s'assurer que le nombre de server redevienne impaire.
Pour ça, il faut :

1. supprimer proprement le server down
2. seulement APRES, supprimer correctement le server 4
3. le server 4 devient agent. On a donc 3 servers et 1 agent.

##### Situation 2 : 6 machines (5 servers + 1 agent), puis 1 server tombe en panne

Si la panne dure, on doit scale up, afin de s'assurer que le nombre de server redevienne impaire.
Pour ça, il faut :

1. supprimer proprement le server down
2. seulement APRES, convertir l'agent en server
3. on a donc 5 servers et 0 agent
