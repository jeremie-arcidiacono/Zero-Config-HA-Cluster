### Investigation k3s : comportement lors de panne

#### Lorsqu'un agent tombe

Lorsque le node agent n'envoie plus de heartbeats :

1. Marqué comme NotReady après X temps (défaut : 40 secondes)
2. Les pods sont re-planifiés ailleurs après X temps (défaut : 5min)

Lorsqu'il redevient disponible (marqué comme Ready), le node reprend son role de manière automatique, mais par contre
les pods déplacés ne reviennent pas.

#### Lorsqu'un server tombe

Chaque server fait tourner sa propre instance de "API Server".

L'API Server sur le node devient évidement indisponible, il faut donc que les autres servers prennent le relais.
Les agents sauront trouver les autres servers via leur load-balancer interne, qui connait la liste des servers.  
Source : https://github.com/k3s-io/k3s/discussions/4488
et https://docs.k3s.io/architecture#how-agent-node-registration-works

> Note sur la Virtual IP :  
> Une Virtual IP (partagé entre tous les servers) dans les 2 cas suivants :
> - Si un nouveau node non-enregistré tente de s'enregistrer sur l'ip d'un server mort
> - Si on tente de communiquer avec le cluster via l'extérieur via kubectl

Les 2 points décrits [dans la section "agent"](#lorsquun-agent-tombe) s'appliquent aussi.
Mais bien avant ces 2 points, c'est le process interne au server etcd/raft qui détecteront la panne (de l'ordre que
1 seconde)

Lorsqu'il redevient disponible, son état etcd se synchronise avec le reste des servers.

#### Recréation automatique d'un server à partir d'un agent ? (= promotion agent->server)

Non, k3s n'est pas capable de faire ça.

Dans le cas d'une perte d'un server qui ne dépasse pas le max supporté (1 server mort sur 3 par exemple),
le cluster restera dans un état fonctionnel mais dégradé, jusqu'à une intervention manuelle.

Le cluster ne va pas tenter de compenser, les autres membres vont "seulement" retenter de joindre le node défaillant.

Si on veut le faire, cela doit se faire depuis l'extérieur du cluster k3s, via antsd.

#### Suppression définitive d'un node après panne, de manière automatique

Ce n'est pas automatique, c'est forcément une action manuelle, comme pour le point précédent.  
Donc antsd doit le couvrir.

#### Exemples de choses configurables :

| Comportement                                 | Paramètre                      | Défaut |
|----------------------------------------------|--------------------------------|--------|
| Fréquence de heartbeat                       | `node-status-update-frequency` | 10 s   |
| Délai avant NotReady                         | `node-monitor-grace-period`    | 40 s   |
| Délai avant éviction des pods d'un nœud mort | `pod-eviction-timeout`         | 5 min  |

Les valeurs etcd sont aussi configurables (intervalle heartbeat Raft, timeout election Raft, etc...)

#### Conclusion

Si la panne est "légère" et courte, antsd peut simplement laisser k3s gérer.

Si la panne est "grave" et longue, antsd intervient comme un humain qui tenterait de ramener le cluster en ligne.
